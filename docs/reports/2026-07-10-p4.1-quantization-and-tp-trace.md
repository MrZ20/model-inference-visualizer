## 结论

W8A8 融合路径并非“完全采集不到”。本次通过外部 monkey patch，在不修改 vLLM 和 vLLM Ascend 源码仓的前提下，精确采集了 Python 可见的融合算子边界：MoE top-k、dispatch 后 INT8 激活及每 Token 动态 scale、GMM1+SwiGLU 输出 scale、GMM2 输入 scale 和最终 BF16 输出。

融合 attention 内部未返回的完整 score/softmax 矩阵仍不可由普通 Python hook 直接获得，后续不得伪造成真实数据。

## 量化路径真实数据

以 layer 3 prefill 为例，每个 rank 都记录了完整的小 scale 张量：

| 边界 | Tensor | rank 0 | rank 1 |
|---|---|---:|---:|
| MoE dispatch 输出 | INT8 hidden states | `[40, 2048]` | `[40, 2048]` |
| dispatch 动态 scale | FP32 per-token scale | `[40]` | `[40]` |
| GMM1+SwiGLU 输出 | INT8 activation | `[40, 256]` | `[40, 256]` |
| SwiGLU 输出 scale | FP32 per-token scale | `[40]` | `[40]` |
| GMM2 输出 | BF16 hidden states | `[40, 2048]` | `[40, 2048]` |

dispatch scale 在两个 rank 上相同，范围为约 `0.04035–0.06299`；这是两侧进入专家计算前共享的量化输入。SwiGLU 后 scale 已随本地专家计算分化：rank 0 约 `0.00078–0.00923`，rank 1 约 `0.00097–0.05730`。网页可以据此直观展示“相同路由输入进入两个 TP 分片，局部计算后中间量化分布不同”。

prefill 共有 5 个 token，每个 token 选择 8 个专家，因此 dispatch 展开为 40 行。top-k ID、权重和 256 项专家 Token count 均按完整值采集，不是抽样猜测。

## TP2 并行证据

两个 rank 各记录 742 个事件。代表性本地分片为：

- QKV projection：逻辑输出 9216，每个 rank 输出 4608；本地 INT8 权重均为 `[2048, 4608]`。
- O projection：逻辑输入 4096，每个 rank 输入 2048；本地权重均为 `[2048, 2048]`。
- MoE router：两边均有 `[256, 2048]` BF16 权重，属于 replicated 计算。
- MoE experts：每个 rank 的 W1 INT8 分片为 `[256, 2048, 512]`，W2 INT8 分片为 `[256, 256, 2048]`。
- layer 3 的 QKV、融合 attention、O projection、router 和 experts 均有 rank 0/1 开始、结束和 duration。

prefill 代表 span：

| 阶段 | rank 0 | rank 1 |
|---|---:|---:|
| QKV projection | 6.46 ms | 6.33 ms |
| fused attention | 6.06 ms | 6.21 ms |
| O projection | 0.86 ms | 1.43 ms |
| MoE router | 3.56 ms | 3.55 ms |
| MoE experts | 9.65 ms | 9.83 ms |

这些 duration 用于展示同一步骤中两条 rank 时间线，不作为性能基准：运行是 eager，且数值摘要会引入同步和采集开销。

运行中 `get_ep_group()` 显示 world size 2，是因为 EP=false 时 MoE 通信组复用了 TP group。前端必须标成“TP2 / MoE communication group”，不能标成独立的 EP2。
