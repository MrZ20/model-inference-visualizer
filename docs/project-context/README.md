# 项目连续性档案

> 作用：这是项目在对话压缩、换会话或长期中断后的**首要恢复入口**。`MEMORY.md` 只保留摘要，不能单独代表完整需求。
>
> 最近更新：2026-07-13

## 为什么存在

本项目已经发生过两类偏离：一是页面组件存在但用户实际无法点击或看见展开结果；二是进度条在动，但正文画面和镜头没有形成连续推理动画。更根本的问题是，对话压缩后只保留了事实摘要，视觉稿为何被选择、用户反复强调的体验、失败验收和纠偏过程没有被完整恢复，后续实现便容易从当前代码反推产品意图。

因此，本目录把长期上下文拆成六类信息：需求来源、总方案、视觉交互契约、当前真实状态、恢复协议和读取顺序。当前实现不是需求来源；已经通过的自动化测试也不能替代用户对产品方向的验收。

## 强制读取顺序

每次开始较大工作，以及每次系统发生上下文压缩后，必须按下列顺序读取：

1. [`CHAT_TIMELINE.md`](CHAT_TIMELINE.md)：用户原话与需求演变。
2. [`MASTER_PLAN.md`](MASTER_PLAN.md)：完整产品方案、数据方案、架构和阶段门。
3. [`VISUAL_INTERACTION_CONTRACT.md`](VISUAL_INTERACTION_CONTRACT.md)：不可随意发挥的视觉与交互约束。
4. [`CURRENT_STATE.md`](CURRENT_STATE.md)：当前分支、数据、实现、测试、验收状态和下一步。
5. [`COMPACTION_RECOVERY.md`](COMPACTION_RECOVERY.md)：压缩后的恢复动作与更新规则。
6. 仓库根目录的 `MEMORY.md`、`TASKS.md`，再检查 `git status` 和 staged diff。

机器与执行者都应以 [`CONTEXT_MANIFEST.md`](CONTEXT_MANIFEST.md) 为清单，不能只读其中一份。

## 信息权威顺序

发生冲突时按以下顺序处理：

1. 用户最新的明确指令；
2. `CHAT_TIMELINE.md` 中已记录的用户明确决定；
3. 本目录的视觉交互契约与总方案；
4. 已接受的 `docs/decisions/` ADR；
5. `CURRENT_STATE.md`、`TASKS.md` 和阶段报告；
6. `MEMORY.md` 摘要；
7. 当前网页或当前代码表现。

如果第 1 项尚未写入仓库，必须先更新本目录再继续实现。不能因为当前代码已经采用某种布局，就把它当成用户原本想要的布局。

## 维护原则

- 用户原话尽量逐字记录；无法证明逐字准确的内容必须标记为“重建摘要”。
- 产品范围或交互决定变化时，更新 `MASTER_PLAN.md` 或视觉交互契约。
- 每个大步骤结束时更新 `CURRENT_STATE.md`，并按用户要求停下等待验收。
- 失败验收不能写成“已完成”；保留问题、根因和恢复条件。
- `MEMORY.md` 继续作为快速摘要，但不再承担完整上下文的职责。
- 不编造被压缩后不可见的历史消息。本目录所称“完整”，指当前能够从用户原话、仓库文档、视觉稿、Git 和报告中审计恢复的完整记录。
